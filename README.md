# [Day 1 종합 실습] 데이터 수집 미니 파이프라인

공개 API 3종을 **비동기로 동시 수집** → **Pydantic v2 스키마 검증** → **CSV·Parquet 저장 및 성능 비교**까지 수행하는 미니 데이터 파이프라인입니다.

- 작성자: 광주캠퍼스 4반 박영서
- 작성일: 2026-07-20

---

## 1. 프로젝트 개요

```
수집(asyncio + httpx) → 검증(Pydantic v2) → 저장(CSV/Parquet) → 성능 비교 → 재로딩 검증 → 예외 처리 시연
```

| 단계 | 내용 | 사용 기술 |
|---|---|---|
| 수집 | 3개 API 동시 호출, 재시도·미러 폴백 | `asyncio.gather(return_exceptions=True)`, `httpx.AsyncClient` |
| 검증 | 필요한 필드만 추출해 타입·범위 확인 | Pydantic v2 `BaseModel`, `Field`, `field_validator` |
| 저장 | 검증 통과분을 두 포맷으로 저장 | pandas, pyarrow |
| 비교 | 쓰기·읽기 시간과 파일 크기 측정 | `timeit` (두 포맷 동일 반복 횟수) |

### 수집 대상 API

| 식별자 | API | 내용 |
|---|---|---|
| `weather` | Open-Meteo | 서울 3일치 시간대별 기온·강수확률 (72행) |
| `country` | Countries.dev | 한국(KOR) 국가 정보 |
| `ip` | ip-api | IP(8.8.8.8) 기반 지역 정보 (차단 시 `ipwho.is` 미러로 자동 전환) |

각 요청은 실패 시 지수 백오프(0.5→1.0초)로 최대 3회까지 재시도하고,
그래도 실패하면 같은 정보를 제공하는 미러 주소로 전환합니다. (`src/collect.py`)

## 5. 실행 결과 요약

### 비동기 수집
3개 API를 `asyncio.gather()`로 동시 호출해 **3/3건 모두 수집 성공**했습니다. 순차 호출이었다면 각 응답 시간의 합만큼 걸렸을 작업이, 가장 느린 한 건의 시간으로 수렴합니다 (실측: 실제 16.70초 / 순차 합산 추정 18.50초).

`ip` API는 작성 환경 네트워크에서 차단되어 3회 재시도가 모두 타임아웃했지만, 미러(`ipwho.is`)로 전환해 최종적으로 수집에 성공했습니다.

### 스키마 검증
72건의 시간대별 예보가 `WeatherRecord`를, 국가·IP 정보가 각 1건씩 `CountryInfo`·`IpInfo`를 통과했습니다(오류 0건). 기온은 -60~60℃, 강수확률은 0~100% 범위로 제한해 API 응답이 비정상일 때 하류로 전파되지 않게 막았습니다.

### CSV vs Parquet 성능 비교

**수집 데이터 (72행)** — 반복 20회 평균

| format | write_ms | read_ms | size_kb |
|---|---|---|---|
| CSV | 0.531 | 0.560 | 5.52 |
| Parquet | 0.799 | 1.007 | 6.05 |

**확대 데이터 (36,000행)** — 동일 조건 재측정

| format | write_ms | read_ms | size_kb |
|---|---|---|---|
| CSV | 79.535 | 26.751 | 2,710.06 |
| Parquet | 10.795 | 1.522 | 10.79 |

→ 36,000행 기준 Parquet이 **읽기 17.6배·쓰기 7.4배 빠르고 파일 크기는 251배 작습니다.**

## 6. 결과 분석 및 의견

### 소량 데이터에서는 CSV가 더 빨랐다
72행 구간에서는 오히려 CSV의 읽기가 빨랐습니다. Parquet은 스키마 메타데이터를 읽고 컬럼 청크를 해석하는 고정 비용이 있어, 데이터가 작으면 이 오버헤드가 이득을 상회합니다. **"Parquet이 항상 빠르다"는 명제는 틀렸고, 손익분기점이 존재합니다.** 실습 자료의 "CSV 대비 10배 빠른 읽기"는 대용량을 전제한 수치임을 실측으로 확인했습니다.

### 파일 크기 251배 차이의 원인
확대 데이터는 `country`·`capital`·`population` 컬럼의 값이 모든 행에서 동일합니다. Parquet은 컬럼형 저장이라 이런 반복 값을 딕셔너리 인코딩으로 압축하지만, 행 기반인 CSV는 같은 문자열을 36,000번 그대로 기록합니다. **데이터 형태가 압축률을 좌우한다**는 점이 드러납니다.

### 벤치마크 워밍업의 필요성
처음 측정했을 때 72행 Parquet 읽기가 184ms로 나왔습니다. 이는 Parquet이 느려서가 아니라 `pyarrow` 엔진의 최초 로딩 비용이 `timeit` 첫 반복에 섞여 들어간 결과였습니다. 측정 전에 쓰기·읽기를 각 1회씩 워밍업하도록 고치자 1.0ms 안팎으로 떨어졌습니다. **측정값이 이상하면 대상보다 측정 방법을 먼저 의심해야 한다**는 점을 확인했습니다.

### 예외 처리는 '넣었다'가 아니라 '동작한다'를 보여야 한다
실제 API 응답이 모두 정상이라 검증 실패 분기가 한 번도 실행되지 않았습니다. 그러면 예외 처리를 작성했다는 사실만 남고 동작한다는 근거는 남지 않습니다. 그래서 파이프라인 [5]단계에서 범위 초과(강수확률 150%)와 타입 오류(기온 `"이십도"`)를 일부러 주입해, 3건 중 2건이 사유와 함께 차단되는 것을 출력하도록 했습니다.

### 재시도와 폴백은 성격이 다른 장치다
재시도만으로는 `ip-api` 차단을 넘지 못했습니다. 재시도는 *일시적* 오류(순간적인 네트워크 오류, 5xx)를 전제한 장치라, 포트가 아예 막힌 *지속적* 차단에는 몇 번을 다시 걸어도 같은 결과입니다. 그래서 재시도(같은 주소, 간격 증가) 위에 미러 전환(다른 주소)을 한 겹 더 두었고, 그 결과 3/3 수집에 성공했습니다.

미러는 응답 키 이름이 달라(`query`↔`ip`, `lat`↔`latitude`) 그대로는 검증에 실패합니다. 스키마를 공급자별로 늘리는 대신 `normalize_ip_payload()`에서 한 형태로 맞춰, **공급자가 늘어도 하류 코드는 그대로**이도록 했습니다.

### 개선 사항
- **증분 수집**: 매 실행마다 전체를 새로 받습니다. 마지막 수집 시각을 기록해 변경분만 가져오면 API 호출량을 줄일 수 있습니다.
- **커버리지 측정**: `pytest --cov=src`로 테스트가 닿지 않는 경로를 확인해야 합니다. 재시도·폴백 경로는 `httpx.MockTransport`로 덮었지만, 아직 전체 커버리지를 수치로 확인하지는 않았습니다.
- **스키마 버전 관리**: API 응답 형태가 바뀌면 검증이 일괄 실패합니다. 스키마에 버전을 부여하고 실패율이 임계치를 넘으면 알림을 보내는 장치가 필요합니다.
- **country·ip 오류 시연 확대**: 예외 처리 시연은 `weather` 모델로만 합니다. 나머지 두 모델도 같은 방식으로 확인하면 검증 근거가 더 완전해집니다.

### 알려진 제약
`ip-api.com`은 작성 환경의 네트워크에서 80·443 포트 모두 차단되어 접근되지 않습니다(재시도 3회 전부 타임아웃). 미러 전환으로 수집 자체는 성공하지만, 실습 지정 주소를 그대로 쓰지 못한 점은 환경 제약입니다. 차단이 없는 네트워크에서는 1회 시도로 `ip-api`에서 바로 수집되며, 어느 쪽에서 받아왔는지는 수집 결과의 `source` 필드로 확인할 수 있습니다.
